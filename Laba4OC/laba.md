## Попередня підготовка

### Словник базових термінів

* **Process:** An executing instance of a computer program, encompassing its code and current activity.
* **PID (Process ID):** A unique numeric identifier automatically assigned to each running task in the OS.
* **PPID (Parent Process ID):** The identifier of the parent process that originally spawned the current process.
* **Terminal (TTY):** The text-based interface environment or virtual console used to interact with the OS shell.
* **Signal:** An asynchronous software notification sent to a process to trigger a specific behavior (e.g., termination or suspension).
* **Parent Process:** A running program that has initiated one or more secondary (child) processes.
* **Child Process:** A sub-process created by a parent program, typically inheriting certain attributes from it.
* **Background Process:** A task running non-interactively behind the scenes, freeing up the terminal for new user inputs.
* **Scheduler:** The kernel subsystem responsible for allocating CPU time to active threads and processes based on priority.

### Відповіді на питання попередньої підготовки

#### 1. Які команди для моніторингу стану процесів ви знаєте. Як переглянути їх можливі параметри?
Для відстеження системних завдань найчастіше використовують такі інструменти:
* **`ps` (Process Status):** Робить статичний "зріз" поточних процесів. Перелік доступних аргументів можна дізнатися через офіційний мануал: `man ps` або швидку довідку `ps --help`.
* **`top`:** Класичний інструмент для моніторингу навантаження у реальному часі. Щоб ознайомитись із його параметрами та гарячими клавішами, можна натиснути `h` безпосередньо під час роботи утиліти, або використати `man top`.
* **`htop`:** Більш сучасна, кольорова та інтерактивна версія `top`. Довідка викликається клавішею `F1` прямо в інтерфейсі.
* **`pgrep`:** Дозволяє фільтрувати та знаходити процеси за їхнім іменем. Деталі синтаксису описані в `man pgrep`.

#### 2. Чи може команда ps у реальному часі відслідковувати стан процесів?
Ні, не може. Утиліта `ps` генерує одноразовий статичний звіт — інформація актуальна лише на ту частку секунди, коли команда була виконана. Для безперервного спостереження слід застосовувати `top` або `htop`.

#### 3. За якими параметрами можливе сортування процесів в команді top? Як переключатись між ними?
Стандартно `top` виводить процеси, відсортовані за рівнем споживання процесорного часу (`%CPU`). Однак доступні й інші критерії:
* `%MEM`: За обсягом використаної оперативної пам'яті.
* `TIME+`: За сумарним часом використання процесора з моменту запуску.
* `PID`: За порядковим номером ідентифікатора.
* `NI`: За рівнем пріоритету (nice).
**Процес перемикання:**
1. Під час роботи `top` натисніть `F` (або `Shift + f`), щоб відкрити меню управління полями.
2. Стрілками на клавіатурі оберіть бажаний параметр (наприклад, `%MEM`).
3. Натисніть клавішу `s`, щоб закріпити цей параметр як критерій сортування.
4. Натисніть `q` або `Esc` для повернення до головного екрана моніторингу.

#### 4. Які команди для завершення роботи процесів ви знаєте?
Для примусової зупинки або надсилання сигналів процесам існують дві базові утиліти:
* **`kill`:** Звертається до процесу за його унікальним номером (`PID`).
    * Стандартний виклик `kill <PID>` надсилає м'який сигнал завершення `SIGTERM`.
    * Жорстка зупинка `kill -9 <PID>` використовує сигнал `SIGKILL`, який миттєво "вбиває" процес.
* **`killall`:** Звертається до процесів за їхньою текстовою назвою, завершуючи всі відповідні екземпляри.
    * Наприклад, `killall chrome` закриє всі процеси браузера. Також підтримує ключі сигналів (наприклад, `-9`).

### 2. Відповіді на питання:

* **Як вивести вміст директорії /proc? Де вона знаходиться та для чого призначена? Охарактеризуйте інформацію про її вміст?**
    Переглянути вміст можна командою `ls /proc`. Це особлива псевдофайлова система, що існує виключно в оперативній пам'яті комп'ютера. Її головна мета — бути комунікаційним мостом між ядром ОС та користувачем. Всередині знаходяться директорії з номерами PID (містять дані про стан конкретного процесу, його пам'ять тощо), а також текстові файли, які репрезентують інформацію про залізо (`/proc/cpuinfo`) чи стан ядра (`/proc/version`).

* **Як вивести інформацію про поточні сеанси користувачів. Якою командою це можна зробити?**
    Базова інформація (хто онлайн, з якого термінала і коли підключився) виводиться утилітою `who`. Більш розширений звіт, що додатково показує рівень завантаженості системи та процеси, які зараз виконують ці користувачі, можна отримати за допомогою `w`.

* **Які дії можна зробити в терміналі за допомогою комбінацій Ctrl + C, Ctrl + D та Ctrl + Z?**
    * `Ctrl + C`: Генерує сигнал `SIGINT`, змушуючи активний процес на передньому плані негайно припинити свою роботу.
    * `Ctrl + D`: Надсилає маркер кінця файлу (EOF). Використовується для закриття поточного сеансу термінала або сигналізує програмам (наприклад, `cat`), що ввід даних з клавіатури завершено.
    * `Ctrl + Z`: Генерує сигнал `SIGTSTP`, який "заморожує" поточний процес і переносить його у фоновий список завдань, повертаючи користувачу контроль над терміналом.

* **Чим відрізняється фоновий процес від звичайного. Де вони використовуються?**
    Звичайний процес працює на передньому плані (foreground), повністю блокуючи термінал, доки не завершить своє виконання або не буде зупинений.
    Фоновий процес (background) працює паралельно і непомітно для інтерфейсу, що дозволяє користувачу продовжувати вводити нові команди. Фоновий режим ідеальний для запуску серверних демонів, тривалого компілювання коду або завантаження великих файлів.

* **Опишіть наступні команди та поясніть що вони виконують – команда jobs, bg, fg.**
    * `jobs`: Друкує список усіх завдань, які зараз прив'язані до поточного вікна термінала, вказуючи їхній індекс та поточний стан (Running/Stopped).
    * `bg` (background): Дає команду призупиненому процесу продовжити своє виконання, але у фоновому режимі (наприклад, `bg %1`).
    * `fg` (foreground): Витягує фонове або заморожене завдання на передній план, знову блокуючи термінал (наприклад, `fg %1`).

* **Якою командою можна переглянути інформацію про запущені в системи фонові процеси та задачі?**
    Для завдань, що належать вашій поточній сесії термінала, використовується `jobs`. Щоб побачити взагалі всі фонові системні процеси (демони), необхідно використовувати глобальні інструменти на кшталт `ps` (наприклад, `ps -e`) або `top`.

* **Як призупинити фоновий процес, як його потім відновити та при необхідності перезапустити?**
    Прямо "заморозити" процес, який вже працює у фоні, неможливо. Спочатку його треба витягнути на екран командою `fg %<номер>`, а вже потім призупинити комбінацією `Ctrl + Z`.
    Відновлення відбувається так:
    * У фоні: вводимо `bg %<номер>`.
    * На передньому плані: вводимо `fg %<номер>`.
    Щоб перезапустити, процес спочатку жорстко закривають (`kill %<номер>`), а потім викликають відповідну команду наново.

## Контрольні запитання

1.  **Яке призначення директорії /proc в системах Linux. Яку інформацію вона зберігає?**
    Це системна директорія, яка монтується в оперативну пам'ять (procfs). Вона слугує "вікном" у ядро операційної системи. Жодних реальних файлів там немає — дані генеруються на льоту під час їх читання. У ній знаходяться папки з інформацією про кожен працюючий процес (аргументи запуску, пам'ять, файлові дескриптори), а також загальні системні файли (інформація про процесор, шини, переривання тощо).

2.  **Як серед будь-яких трьох процесів динамічно визначати, який з них в поточний момент часу використовує найбільший обсяг пам'яті? Який відсоток пам’яті він споживає від загального обсягу?**
    Необхідно запустити системний монітор `top` (або `htop`). В інтерфейсі `top` треба натиснути `F` для виклику меню сортування, вибрати пункт `%MEM` та натиснути `s`. Найвимогливіший до пам'яті процес опиниться на першому рядку. Відсоток його споживання відображатиметься у колонці `%MEM` і розраховується відносно загального обсягу фізичної оперативної пам'яті (без урахування swap).

3.  **Як отримати ієрархію батьківських процесів в системах Linux? Наведіть її структуру та охарактеризуйте.**
    Дерево процесів можна вивести командами `ps -ef --forest` або `pstree`. 
    
    Архітектура побудована за принципом родоводу. На самій вершині завжди стоїть кореневий процес (`init` або `systemd`) з ідентифікатором PID 1. Усі інші процеси є його прямими чи непрямими "нащадками". Створення нового процесу відбувається шляхом копіювання батьківського (виклик `fork()`) та подальшої заміни коду (виклик `exec()`). Якщо батьківський процес гине, система "удочеряє" його фонові процеси до PID 1.

4.  **Чим відрізняється команда top від ps?**
    Команда `ps` працює як фотоапарат: вона робить знімок стану процесів у ту мілісекунду, коли була запущена, виводить текст і завершується. Натомість `top` працює як відеокамера: це інтерактивний додаток, який "висітиме" в терміналі й оновлюватиме статистику щосекунди, дозволяючи моніторити систему у динаміці.

5.  **Які додаткові можливості реалізує htop в порівнянні з top?**
    Утиліта `htop` пропонує значно вищий рівень юзабіліті:
    * Графічне візуальне відображення навантаження на кожне ядро CPU та пам'ять.
    * Підтримка кліків мишкою.
    * Горизонтальна та вертикальна прокрутка всього масиву процесів.
    * Наявність вбудованого меню сигналів (F9), яке дозволяє завершувати процеси без ручного введення PID.
    * Зручна ієрархічна структура процесів (режим дерева доступний натисканням однієї кнопки).

6.  **Опишіть компоненти вашої мобільної ОС для здійснення моніторингу запущених в системі процесів?**
    *На прикладі ОС Android:* Андроїд використовує власне середовище для моніторингу на базі ядра Linux. Основну роль тут відіграє системна служба `Activity Manager`. Доступ до статистики процесів користувач може отримати через меню "Параметри розробника -> Статистика процесів / Служби, що виконуються". Крім того, на рівні ядра працює механізм LMK (Low Memory Killer), який самостійно аналізує важливість процесів та "вбиває" фонові додатки, якщо системі бракує ОЗП.

7.  **Чи підтримує Ваша мобільна ОС термінальне керування роботою процесів, опишіть як саме.**
    Так. На Android можна встановити сторонній емулятор термінала, наприклад, `Termux`. Оскільки ядро системи лінуксоподібне, там працюють базові утиліти `ps`, `top`, `kill`. Однак існує серйозне обмеження: без наявності прав суперкористувача (Root) ви зможете керувати лише тими процесами, що запущені в ізольованому середовищі самого емулятора, і не матимете доступу до загальносистемних процесів.

8.  **Чи можливо поставити сторонні програмні засоби, що дозволяють організувати управління та моніторинг роботою процесів у Вашому мобільному телефоні. Коротко опишіть їх.**
    Існує безліч утиліт для Android, що забезпечують зручний моніторинг:
    * **Task Managers / System Monitors:** Додатки, які графічно показують список активних програм та відсоток навантаження на процесор (наприклад, OS Monitor). Вони можуть містити кнопку примусового завершення додатків.
    * **Greenify:** Специфічний інструмент управління фоновими процесами, який примусово вводить додатки в режим глибокого сну (hibernation) для економії заряду.
    * **Вбудовані утиліти безпеки:** Більшість сучасних прошивок (MIUI, OneUI) мають системні менеджери типу "Оптимізація" або "Догляд за пристроєм", які автоматично аналізують та очищують запущені процеси в один клік.

We significantly enhanced our understanding of job control mechanisms, practicing how to fluidly shift active tasks between the foreground and background using the `fg` and `bg` utilities, while effectively suspending them via keyboard shortcuts like `Ctrl+Z`.

Moreover, this session emphasized the critical role of software signals. By issuing termination commands such as `kill` and generating interrupts like `SIGINT`, we learned to safely halt unresponsive or unneeded tasks. Analyzing the parent-child dependencies through hierarchical tree views solidified our comprehension of how the operating system structure evolves dynamically from the initial boot sequence. These competencies are indispensable for optimizing system performance and troubleshooting software behavior in any Unix-based environment.
