### Завдання 1. Теоретичні питання

**1. Основні функції планувальника завдань та порівняння Windows і Linux.**
Планувальник завдань — це системний компонент (або фонова служба), який дозволяє автоматично запускати програми, скрипти або команди у заздалегідь визначений час або при настанні певних подій.
*   **Основні функції:** автоматизація рутинних процесів (наприклад, резервне копіювання), очищення тимчасових каталогів, оновлення системи, моніторинг ресурсів та розсилка звітів.
*   **Windows (Task Scheduler):** Має потужний графічний інтерфейс (GUI). Дозволяє запускати завдання не лише за часом, але й за складними тригерами (вхід користувача в систему, блокування ПК, запис певного івенту в журнал подій). Формат зберігання завдань базується на XML-файлах.
*   **Linux (Cron / Systemd):** Орієнтований на командний рядок (CLI). Cron працює переважно з прив'язкою до точного часу та дати. Налаштування зберігаються у звичайних текстових файлах (`crontab`), що робить їх ідеальними для перенесення та використання в серверних середовищах.

**2. Принципи роботи з планувальником Cron.**
Cron — це демон (фоновий процес `crond`), який щохвилини перевіряє конфігураційні файли (кронтаби) на наявність завдань, час виконання яких вже настав.
*   **Налаштування:** Здійснюється через редагування файлу `crontab` конкретного користувача за допомогою команди `crontab -e`.
*   **Синтаксис:** Рядок завдання складається з 6 полів: `[Хвилина] [Година] [День_місяця] [Місяць] [День_тижня] [Команда]`.
*   **Альтернативи Cron:**
    *   **Anacron:** На відміну від Cron, не вимагає, щоб система працювала цілодобово. Якщо комп'ютер був вимкнений під час запланованого завдання, Anacron виконає його після увімкнення.
    *   **Systemd Timers:** Сучасна та потужніша альтернатива. Дозволяє встановлювати залежності (запускати завдання тільки якщо активна мережа), має краще логування та мілісекундну точність.
    *   **at:** Використовується для одноразового виконання команди у майбутньому (Cron призначений для періодичних завдань).

### Завдання 2. Практичне використання Cron (Опис методики)

*Примітка: На жаль, під час виконання практичної частини виникли технічні проблеми зі службою планувальника на віртуальній машині. Тому, відповідно до умов завдання, наводжу детальну методику налаштування обраних задач.*

Як приклад задачі візьмемо **резервне копіювання конфігурацій** (створення архіву). Команда для виконання: `tar -czf /backup/config.tar.gz /etc`.
Для додавання цих завдань необхідно відкрити редактор планувальника командою: `crontab -e`.

**1. Виконання задачі в чітко визначений час (наприклад, о 08:30):**
Методика: У полі хвилин вказуємо 30, у полі годин — 8. Інші параметри залишаємо як `*` (кожного дня).
*Запис у crontab:* 
`30 8 * * * tar -czf /backup/config.tar.gz /etc`

**2. Виконання однієї й тієї ж задачі двічі в день (о 09:00 та 21:00):**
Методика: Використовуємо кому для перерахування годин у другому полі.
*Запис у crontab:* 
`0 9,21 * * * tar -czf /backup/config.tar.gz /etc`

**3. Виконання задачі тільки в будні у чітко визначений проміжок часу (кожної години з 8 до 18):**
Методика: У полі годин використовуємо дефіс для вказівки діапазону `8-18`, а в полі днів тижня вказуємо `1-5` (з понеділка по п'ятницю).
*Запис у crontab:* 
`0 8-18 * * 1-5 tar -czf /backup/config.tar.gz /etc`

**4. Виконання задач з використанням спеціальних макросів:**
Методика: Cron підтримує спеціальні слова-замінники для стандартних розкладів замість написання цифр.
*   Раз у рік: `@yearly tar -czf /backup/config.tar.gz /etc`
*   Раз у місяць: `@monthly tar -czf /backup/config.tar.gz /etc`
*   Раз у день: `@daily tar -czf /backup/config.tar.gz /etc`
*   Щогодини: `@hourly tar -czf /backup/config.tar.gz /etc`
*   При вмиканні (після перезавантаження): `@reboot tar -czf /backup/config.tar.gz /etc`

### Завдання 3. Альтернативний планувальник задач (Systemd Timers)

Як альтернативу Cron було обрано **Systemd Timers**. Це сучасний стандарт для більшості дистрибутивів Linux (Ubuntu, Debian, CentOS).

**Методика налаштування (для виконання задачі щодня о 08:30):**

1. На відміну від Cron, Systemd вимагає створення двох файлів. Спочатку створюється файл служби (Service), який описує, *що* робити.
Створюємо файл `/etc/systemd/system/backup.service` з наступним вмістом:

    [Unit]
    Description=Backup /etc directory

    [Service]
    Type=oneshot
    ExecStart=/bin/tar -czf /backup/config.tar.gz /etc

2. Далі створюється файл таймера (Timer), який описує, *коли* це робити.
Створюємо файл `/etc/systemd/system/backup.timer` з наступним вмістом:

    [Unit]
    Description=Run Backup Daily at 08:30

    [Timer]
    OnCalendar=*-*-* 08:30:00
    Persistent=true

    [Install]
    WantedBy=timers.target

*(Параметр Persistent=true працює як Anacron: якщо ПК був вимкнений о 08:30, завдання виконається одразу після його увімкнення).*

3. Для активації планувальника виконуються команди:
`sudo systemctl daemon-reload`
`sudo systemctl enable --now backup.timer`

### Glossary
*   **Task Scheduler** - Планувальник завдань (система для автоматичного запуску процесів за розкладом).
*   **Cron job** - Завдання Cron (конкретна команда або скрипт, що виконується за розкладом).
*   **Crontab (Cron table)** - Таблиця Cron (конфігураційний файл, що містить список завдань та час їх виконання).
*   **Daemon** - Демон (фоновий процес у UNIX-подібних системах, який працює без прямої взаємодії з користувачем).
*   **Backup** - Резервне копіювання (процес створення копії даних для їх збереження та відновлення у разі втрати).
